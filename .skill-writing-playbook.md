# Skill Writing Playbook

Compiled from Claude Code docs + Thariq's "Lessons from Building Claude Code: How We Use Skills".

## Core Principles

### Description is the model
The `description` field is NOT a summary. It's a trigger description — Claude decides "is there a skill for this?" based on it. Write it as: when to use this skill, with specific keywords users would naturally say. The description goes into the skill listing Claude builds at session start.

```yaml
# BAD
description: "React component patterns"

# GOOD
description: "Enforce our React component conventions. Use when creating new components, reviewing React code, or when the user asks about our design system patterns."
```

### Don't state the obvious
Claude already knows a lot. Focus on information that makes Claude think differently from its defaults. "How we use React" is boring. "How we use React at Anthropic by starting with a custom hook improving Claude's design decisions" is valuable.

### Build a Gotchas section
The highest-signal content in any skill. Capture common failures, footguns, edge cases — things that only come from experience using the skill. Update this over time.

## Structure

### Progressive disclosure via file system
Don't put everything in one SKILL.md. Use the directory structure:

```
my-skill/
├── SKILL.md              # Core instructions + triggers
├── references/            # On-demand deep content
│   ├── gotchas.md        # Edge cases, common failures
│   └── detailed-api.md   # Reference material
├── scripts/              # Executable tools
│   └── validate.sh
├── templates/            # Output templates
│   └── report.md
└── examples/             # Expected output formats
    └── sample.md
```

The key insight: the file system IS context engineering. Claude reads files at appropriate times. Only loads what it needs.

### Two content types
- **Reference content** — knowledge/conventions Claude applies to current work. Runs inline with conversation context.
- **Task content** — step-by-step instructions for specific actions. Often better with `disable-model-invocation: true` so only manual invocation.

## Writing Tips

### Avoid railroading
Give Claude information + flexibility. Too-specific instructions prevent adapting to the situation. State what Claude needs to know, not exactly what to do with it.

### Think through setup
Some skills need context from the user. Store setup in a config file. The skill checks if config exists — if not, Claude asks the user. Use structured questions for multiple choice.

### Store scripts & generate code
Give Claude composable functions, not just instructions. A data science skill with helper functions lets Claude generate scripts on the fly for complex analysis. Code > prose for capability.

### Memory & storing data
Skills can include memory via JSON/text files in the skill directory. Example: `state-log.json` tracks what a skill has done, so next run it knows what changed.

### Don't forget hooks
Skills can include on-demand hooks activated only when the skill is called. Useful for safety (blocking dangerous commands) or focus (blocking edits while debugging).

## Frontmatter Fields

| Field | Purpose |
|---|---|
| `name` | Slash command name, unique identifier |
| `description` | Trigger description — when to use this skill |
| `argument-hint` | Hint for autocomplete: `[issue-number]` |
| `disable-model-invocation` | `true` = manual only, Claude won't auto-trigger |
| `hide-from-menu` | `true` = hidden from `/` menu |
| `related_skills` | (Hermes extension) skills to load alongside |
| `metadata.category` | Grouping for discovery |
| `metadata.tags` | Keywords for search |

## Dynamic Context

| Variable | Purpose |
|---|---|
| `$ARGUMENTS` | All arguments passed to the skill |
| `$ARGUMENT[n]` | Specific argument by index |
| `$CLAUDE_SESSION_ID` | Current session ID |
| `$CLAUDE_WORKSPACE` | Path to skill directory |

## Distribution

- Small teams: check skills into `.claude/skills` in repo
- Larger orgs: plugin marketplace with review process
- Every skill in repo adds to context window — be selective
- Make sure there's review before release (easy to create bad/redundant skills)

## Composing Skills

Skills can reference each other by name. The model will invoke related skills if installed. Dependencies aren't natively built in, but `related_skills` frontmatter + natural cross-references work.

## Measuring

Simple feedback loop: ask "did this skill do what you expected?" at end of session. Use answers to iterate on skill quality.

## Skill Categories (from Anthropic's internal use)

2. Product Verification — testing/verifying code works correctly
3. Data Fetching & Analysis — connecting to data stacks

5. Code Scaffolding & Templates — framework skeletons
6. Code Quality & Review — enforcing standards beyond linters
7. CI/CD & Deployment — build, ship, deploy
8. Runbooks — symptom → investigation → report
9. Infrastructure Operations — routine maintenance & ops

## For SEAL Skills Specifically

Applying these patterns to our security knowledge graph:

1. **Each SKILL.md should have a strong description** with domain-specific keywords (not just "security framework")
2. **Gotchas section** in each skill for common pitfalls in that security domain
3. **Progressive disclosure** already in place — references/ loaded on demand
4. **Cross-references** via `related_skills` in frontmatter (already done)
5. **INDEX.md** as the MOC trigger table (already done)
6. **Scripts in references/** for validation, checking, or generating security reports
7. **Memory** — track quiz scores, training completion across sessions
8. **Don't railroad** — give security framework guidance, not rigid steps
